Network Protocols for Recon

Published May 17, 2026 · by 7heKnight

Network & Protocol Recon Nmap

Recon is 80% of an engagement. You can't enumerate what you can't read, so this post is a fast refresher on the protocols that matter and how to pull signal out of them.

The protocol stack

Bottom-up, the layers you'll touch most:

  • Link: Ethernet, ARP (spoofing & MITM).
  • Internet: IP, ICMP (host discovery).
  • Transport: TCP, UDP (port scanning).
  • Application: HTTP, DNS, SMB, SSH, FTP, SNMP.

TCP vs UDP

TCP is a phone call (handshake, ordered, reliable). UDP is a postcard (fire-and-forget). Both leak information.

The TCP 3-way handshake (SYN → SYN/ACK → ACK) is what makes a SYN scan work: send the SYN, read the reply, never finish — fast and relatively quiet.

Recon in practice

# Host discovery + top ports, service/version detection
$ nmap -sS -sV -O --top-ports 1000 10.10.10.0/24

# UDP is slow but DNS/SNMP/TFTP hide here
$ nmap -sU --top-ports 50 10.10.10.5

High-value services to flag immediately:

  • 21/FTP, 22/SSH — anonymous login & weak creds.
  • 53/DNS — zone transfers (dig AXFR).
  • 139,445/SMB — null sessions, shares, EternalBlue-era bugs.
  • 161/SNMP — public community strings dump everything.

Reading the traffic

When a scan isn't enough, capture and inspect. A Wireshark display filter like tcp.flags.syn == 1 && tcp.flags.ack == 0 isolates connection attempts; http.request surfaces every web call. Learn to read packets and the network stops being a black box.
